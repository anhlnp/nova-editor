First real browser audit of nova editor (user-run, 2026-06-27), Part 1 (Keep K1–K10) of doc/qa-checklist.md. **Gate 0 FAILED** — Phase F must NOT be flipped ✅; it is 🔴. This validates the strategic pivot ([[project-nova-strategic-pivot]]): broad logic-complete-but-unverified work hid a fundamental perf bug.

**ROOT CAUSE (confirmed in code) — Phase F deserialize-per-edit.** Every discrete edit goes useDocumentWrite → applyExternalSchema → bumps `canvasSyncToken` → `<CanvasSync>` (page.tsx L383) → applySchemaToCanvas → **full `deserialize` of the whole Craft Frame**. On a page with a Hero (composite, many children), each node's RenderNode (ADR-017) runs getComputedStyle/getBoundingClientRect → every edit re-measures the whole tree ⇒ severe lag + edits appear "not to work" (they apply but the slow full remount loses feedback/state). This is exactly **ADR-042 Risk #3** (deserialize-on-every-edit I9) which was flagged but whose mitigation (scope-to-subtree / commit-on-pointer-up / in-place) was NEVER implemented. **Fix = P0, Major-by-risk/Opus**, must come before anything else (likely also fixes K6 "edits don't work").

**Gate 0 results:** ✅ = K1 draft, K4.1 tab-position, K8 breadcrumb, K9 inline-text(navbar; other blocks untested), K10 selection-survives, K2.1 canvas-render(blocks untested). ❌/⚠️ (the interactive core): K3.1/K3.2 lag on select/empty-click + deleted node still in selection; K6.1–6 ALL discrete edits on hero dead; K2.2 add-page FREEZES canvas (+ missing per-page settings "..." menu like webstudio); K5 palette Delete has no undo + palette keyboard-nav to off-screen items broken; K7 Publish does nothing + no message re: Vercel token.

**Fix backlog from Gate 0:** P0 Phase F perf (Major/Opus) → P1 pages-freeze (Minor) + re-confirm K6 after perf fix → P2 delete-undo (Minor) + publish-silent/token-feedback (Patch/Minor).

**P0 PROGRESS (2026-06-27, ADR-043 — logic done, runtime 🟡):** Implemented targeted canvas projection. New `applyNodePatchToCanvas(actions, ops)` + `NodePatchOp` in CraftProvider; store `canvasSyncOp: {kind:"full"|"patch"}` + `applyNodePatch`; `<CanvasSync>` branches (patch→per-node `history.ignore().setProp`, full→deserialize as before); app hook `useNodePatch()` (setNodeProps Document write + same updater projected to Craft in place). Re-pointed RightPanel props + StylePanel classOverrides `mutate` (the 2 biggest). Document stays authoritative (I2); structural + free-position(2-node) keep full deserialize. Typecheck 8/8, editor 185 + app 284 tests green. **P0 step (1) DONE (2026-06-27):** re-pointed ALL single-node canvas edits to useNodePatch — AlignBar self-align, RenderNode arrow-nudge + resize-commit, LayoutOverlay gap-commit (transient drag previews unchanged; commit-on-pointer-up now targeted). Structural edits (move/insert/delete via cmdDropRelative/cmdInsertBlock/removeNode) + free-position(2-node) keep full deserialize — correct. Only **rename (_novaName)** left on full-deserialize — DEFERRED to G1 (it's prop-drilled through the 2110-line LeftPanel monolith; rare + visually trivial; G1 split cleans it up). Typecheck 8/8, editor 185 + app 284 green.
**P1 pages-freeze (K2.2) — investigated 2026-06-27:** Static analysis found NO infinite loop — pure page ops (createPage/addPage in operations/pages.ts) are clean, schemaToNodes handles empty pages (ROOT + empty nodes[]), PagesPanel doesn't loop, RightPanelWithSync selection-sync doesn't loop (shouldClearCraftSelection only mirrors clears). Conclusion: the "freeze" on add-page/page-switch is most plausibly the SAME O(N) render-storm class as K3 (changing activePage → `<Frame key={activePage}>` full remount + previously every RenderNode re-rendered via the selectedNodeIds array sub) — which the P0/K3 fixes already reduce. Made ONE concrete safe fix: clear uiStore selection on activePage change (editor page.tsx effect on [activePage]) — the old page's selected node no longer exists in the new canvas; fixes stale cross-page selection (matches user's K3.2 note "node cũ vẫn select"). Typecheck 8/8. NEEDS BROWSER QA to confirm whether the freeze persists after P0/K3; if it does, next suspect = Frame full-remount cost on page switch (would need repro to optimize). Did NOT pile on speculative fixes (verified-depth).

**K3/G7 FIX (2026-06-27, perf core done):** Root cause of select/click lag = every RenderNode subscribed to the whole `selectedNodeIds` ARRAY (uiStore) → ALL N nodes re-rendered on any select/clear (O(n), worst on Hero). Fixed with a SCOPED boolean sub `useUIStore(s => s.selectedNodeIds.includes(id))` → only affected nodes re-render. Also gated `<AlignBar/>` to `{isActive && …}` (was rendered in every node; it self-resolves+returns null otherwise, but gating drops N idle instances + N zoom subs). Craft `isActive` collector + `useNode` were already scoped (not culprits). Typecheck 8/8, app 284 green. **DELIBERATELY NOT DONE (deferred):** the full RenderNode→`<NodeChrome>` file-split (move zoom/leftSection/rightPanelOpen subs + measurement effects into a child mounted only when active/hover, so inactive nodes are inert on zoom/panel-toggle). Reason: big hot-path structural change, unit-tests can't verify it (DOM/Craft integration, no RenderNode tests), env is typecheck-only → violates verified-depth principle. The scoped-sub fix already removes the O(n) select/click lag (the reported K3); the zoom/panel-toggle re-render is a lesser, unreported issue. Do the full split when browser QA exists. Browser QA still gates flip to ✅ (confirm K3 + K6 lag gone).

**STRATEGIC REPRIORITIZATION (user, 2026-06-27 — refined feedback promised next prompt):** Pivot from running all G1–G12 parity to a single perfect demo flow. MVP demo MUST be: **active user flow → enter prompt → describe idea → see result** (AI-prompt-driven). Also: (1) **contrast/theme is bad** in both dark+light, no app-wide theme support — fix; (2) **AI must NOT impose templates/style** on the user — serve real intent; work at a minimal "universal" level so the user feels heard, steer toward registered styles, and **clearly state when something isn't supported** (transparency about demo limits); (3) fundraising needs: one complete (even partly mock) end-to-end smooth flow + full product vision + honest limits/roadmap. **How to apply:** when the refined feedback lands, rewrite the MVP plan around ONE AI-prompt flow + perf P0 + theme/contrast + transparency, NOT broad parity. See [[project-nova-editor-redesign]].
